1. LA BASE DE DATOS: DE TEXTO A UN ARCHIVO BINARIO REAL (SQLITE)
------------------------------------------------------------------------
¿Qué hicimos?
Teníamos un archivo llamado "Base de datos.dbml" que era simplemente un 
esquema de texto (un dibujo de cómo debían ser las tablas). Lo que hicimos
fue "traducir" ese texto a código real (SQL) y usarlo para generar un 
archivo llamado "database.sqlite".

Problema que solucionamos:
Hubo un momento donde el sistema daba un error: "SQLITE_NOTADB: file is 
not a database" (El archivo no es una base de datos). Esto pasó porque 
por accidente se guardó texto de programación directamente dentro del 
archivo .sqlite en lugar de generar el archivo binario correcto. 
Lo solucionamos usando un comando de la terminal para "fabricar" la base 
de datos real desde cero.

Ejemplo de cómo se ve el código SQL que crea una tabla:
```sql
CREATE TABLE IF NOT EXISTS "Productos" (
  "id" INTEGER PRIMARY KEY AUTOINCREMENT,
  "nombre" TEXT NOT NULL,
  "tipo" TEXT NOT NULL,
  "precio" REAL NOT NULL,
  "stock" INTEGER DEFAULT 0
);
```
Explicación para defender: "Creamos las tablas estrictas en la base de 
datos. Por ejemplo, la tabla 'Productos' guarda el nombre en texto (TEXT)
y el precio en números con decimales (REAL). Además le indicamos que si no
le pasamos el stock, por defecto (DEFAULT) será cero."

2. ARQUITECTURA: LOS CONTROLADORES (BACKEND Y FRONTEND)
------------------------------------------------------------------------
¿Qué hicimos?
Separamos las cosas. Antes, los botones y las vistas ("Frontend") hacían 
todo el trabajo. Ahora, creamos "Controladores". 
- Frontend: Tiene 14 controladores (uno por cada vista) que preparan la 
  información antes de enviarla.
- Backend: Tiene 14 controladores espejo que reciben esa información, la
  revisan de nuevo (por seguridad) y la guardan en la base de datos.

¿Por qué es importante?
Porque si alguien intenta hackear la página web (Frontend), el servidor 
(Backend) tiene sus propios controladores que frenan cualquier dato malo.

Ejemplo de Controlador en el Backend (VentasController.ts):
```typescript
export const registrarVenta = async (req: Request, res: Response) => {
    const orden = req.body;
    
    // Validar que la orden tenga items
    if (!orden.items || orden.items.length === 0) {
        return res.status(400).json({ error: "La orden está vacía." });
    }

    // (Aquí va el código que guarda en la base de datos)
    res.json({ mensaje: "Venta registrada con éxito" });
};
```
Explicación para defender: "Usamos el patrón MVC (Modelo-Vista-Controlador).
Los controladores son como los 'gerentes'. Cuando el cajero le da al botón
'Vender' en la Vista, la orden viaja al Controlador del Backend. Este 
revisa (if !orden.items) si la orden está vacía. Si está vacía, devuelve
un error 400 (Bad Request). Si todo está bien, lo guarda."

3. VALIDACIONES ROBUSTAS (CUMPLIENDO EL ENUNCIADO)
------------------------------------------------------------------------
¿Qué hicimos?
Aplicamos reglas de negocio estrictas. Por ejemplo, si alguien vende un
"Botellón Lleno", el sistema tiene que validar si hay botellones físicos
en el inventario. Si solo es una "Recarga", no resta botellones.

Ejemplo de código de validación:
```javascript
const items = orden.items;
for (const item of items) {
   if (item.cantidad <= 0) {
       throw new Error("La cantidad debe ser mayor a cero.");
   }
   if (item.tipo === 'Botellón Lleno' && stockBotellones < item.cantidad) {
       throw new Error("No hay suficientes botellones físicos.");
   }
}
```
Explicación para defender: "No usamos librerías externas pesadas para 
validar. Hicimos validaciones nativas con condicionales 'if' para que el
código sea ligero y fácil de leer. Recorremos cada producto comprado,
verificamos que no compren 'cantidades negativas' y aplicamos la regla de
oro: solo restar stock físico si de verdad están comprando un envase."

4. IMPLEMENTACIÓN DE LA API EXTERNA (TASA DEL DÓLAR EN VIVO)
------------------------------------------------------------------------
¿Qué hicimos?
Conectamos nuestro sistema con una API real y pública (ve.dolarapi.com)
para traer el precio del dólar (BCV y Paralelo) en tiempo real al 
Dashboard del Administrador. 

¿Cómo funciona?
Usa una función "fetch" (que significa buscar) para ir a internet, traer
los datos y mostrarlos. Además, se actualiza sola cada 5 minutos.

Ejemplo de código de la API:
```javascript
const fetchTasas = async () => {
  try {
    // 1. Va a internet a buscar la tasa
    const res = await fetch('https://ve.dolarapi.com/v1/dolares');
    
    // 2. Convierte la respuesta a formato JSON (entendible por la PC)
    const data = await res.json();
    
    // 3. Lo guarda en nuestra variable "tasas" para mostrarlo
    tasas.value = data;
  } catch (err) {
    console.error('Error al obtener tasa del dólar');
  }
};
```
Explicación para defender: "Implementamos una conexión asíncrona (async) a
una API REST externa. Usamos 'await' para decirle al sistema: 'espera a 
que el internet responda antes de continuar'. Agregamos un bloque 
'try/catch' para que si el internet se cae, el sistema no explote, sino
que simplemente muestre un mensaje de 'Error de conexión' en la tarjetita
del dólar."

5. SOLUCIÓN AL CRASH DE LA VISTA DE REPORTES
------------------------------------------------------------------------
¿Qué pasó?
Cuando hacías clic en "Reportes", la pantalla se ponía en blanco y salía un 
error: "Cannot read properties of undefined".
Esto pasaba porque el Frontend le pedía datos al Backend, el Backend 
respondía "Todo bien (Código 200)", pero le mandaba datos incompletos 
porque la base de datos real de reportes aún no está lista. El Frontend,
confiado, intentaba leer datos que no existían y explotaba.

¿Cómo lo arreglamos?
1. Le dijimos al Backend que, por ahora, responda con un "Error 501 
   (No implementado)" en vez de un "200 (Todo bien)".
2. Le pusimos un escudo al Frontend: "Si los datos vienen incompletos, lanza
   un error tú mismo".
   
Ejemplo de la solución (El escudo):
```javascript
const balanceData = resBalance.data;

// Si la información viene sin los "estado_resultados", detente.
if (!balanceData || !balanceData.estado_resultados) {
    throw new Error("Datos incompletos desde el backend");
}
```
Explicación para defender: "Aplicamos 'Programación Defensiva'. En lugar 
de confiar ciegamente en lo que responde el servidor, el Frontend primero 
verifica si la estructura de los datos es la correcta. Si no lo es, activa 
el bloque 'catch' y carga datos de prueba (Mock Data) para que el usuario 
nunca vea una pantalla en blanco o rota."

6. DOCUMENTACIÓN Y COMENTARIOS EN EL CÓDIGO
------------------------------------------------------------------------
¿Qué hicimos?
Por último, inyectamos comentarios en los 14 archivos de las vistas 
(.vue) para que cualquier programador nuevo que entre al proyecto sepa 
exactamente qué hace cada archivo sin tener que leer todo el código.

Ejemplo:
```html
<!-- VISTA: Dashboard Administrativo | Propósito: Mostrar métricas 
principales, ventas del periodo, estado del inventario... -->

<script setup>
// [LÓGICA DEL COMPONENTE] Inicialización de estado reactivo...
</script>

<style scoped>
/* [ESTILOS] Reglas CSS exclusivas de esta vista para evitar colisiones */
</style>
```
Explicación para defender: "Buscamos que el código sea 'Auto-documentado'.
No sobrecargamos el código con comentarios inútiles línea por línea, sino
que agregamos bloques estratégicos en las 3 zonas clave de Vue: El HTML 
(Template), el Javascript (Lógica) y el CSS (Estilos). El atributo 'scoped'
en el estilo garantiza que si pinto un botón de rojo aquí, no se pinte de 
rojo en otra pantalla por accidente."
